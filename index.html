Activity与Fragment

Acitivt四种启动模式

启动模式一共有四种，分别是standard、singleTop、singleTask和singleInstance。

standard

standard模式是系统默认的启动模式，在该模式下，每次启动活动，系统就会将该活动添加到返回栈中，而不会考虑返回栈中的情况。例如当活动A处于栈顶时，我们再次启动一次活动A时，系统仍会将一个相同的活动添加到返回栈中。

singleTop

singleTop模式解决了标准模式下的问题，当处于栈顶的活动启动本身的时候系统不会再创建一个新的实例，不会向返回栈中添加新的活动。

singleTask

singleTop模式避免了在栈顶的活动被重复创建的问题。在singleTask模式下，当启动活动A时，系统会首先检查返回栈中是否用活动A的实例，若没有则重新创建，若有则直接使用该实例，并将返回栈中活动A之前的活动移除栈。

排序算法简介

选择排序算法

对于一组给定待排序数组，第一轮遍历数组后将最小元素与第一个数进行交换，第二轮从第二个元素开始遍历，将最小值与第二个元素进行交换，重复进行上述过程，，直至待比较的元素只有一个时为止。

插入排序算法

对于一组给定的待排序数组，初始时将第一个数看作已排序序列，将其余元素看作未排序序列，接着从第二个元素开始遍历，依次将该值插入到前面的已排序序列，直到最后一个元素插入到有序序列之中。

冒泡排序算法

对于给定的包含n个元素的待排序数组，从第一个元素开始对相邻的两个元素进行比较，当前面的元素大于后面的元素时，交换二者位置，进行一轮比较后，n个元素中的最大值将置于第n-1位，然后对剩余（n-1）个元素进行第二轮比较，重复该过程直至剩下一个元素为止。

归并排序

利用递归与分治技术将数据序列划分为越来越小的半子表，再对半子表进行排序，最后再用将排好序的半子表合并为越来越大的有序序列。

Activity的生命周期

返回栈

安卓使用任务(Task)来管理活动，一个任务就是一组存放在栈里的活动的集合，这个栈被称为返回栈。由于栈是具有后进先出的特性，每当启动一个新活动时，这个活动就会入栈，处于栈顶的位置，当按下返回键或者finish()方法时处于栈顶的活动就会出栈，上一个活动重新处于栈顶位置，系统将该活动呈现给用户。

活动状态

每个活动在其生命周期中最多可能会有四种状态。

  • 运行状态。
    即活动处于栈顶的状态，系统将该活动呈现给用户，这时的活动就是处于运行状态。
  • 暂停状态。
    暂停是一种什么状态呢？当呈现在我们面前的界面上出现了一个对话框时，对话框并未占满屏幕，原活动还处于一种可见的状态，这时活动即是暂停的。即当活动不处于栈顶，但是还在可见的状态。
  • 停止状态
    当一个活动不在栈顶，并且处于完全不可见的状态，就进入了停止状态。
  • 销毁状态
    当一个活动被移出返回栈时，该活动即进入了销毁状态。

当手机内存不足时，系统会更倾向于回收不在栈顶的活动，而保留栈顶或者可见的活动，这样会有更好的用户体验。

抽象类与接口

如果一个类中包含抽象方法，那么这个类就是抽象类。在Java语言中，可以把类或者类中的某些方法声明为abstract来表示一个类是抽象类。接口是指一个方法的集合，接口中的所有方法都没有方法的具体实现，在Java中，接口是通过关键字interface来实现的。

抽象类和接口都是支持抽象类定义的两种机制。但是二者存在很大的区别。

只要包含一个抽象方法的类就必须被声明为抽象类，抽象类可以声明方法的存在而不去实现它，被声明为抽象的方法不能包含方法体。在实现时，必须包含相同或更低的访问级别（public、protected、private）。抽象类在使用过程中不能被实例化，但是可以创建一个对象使其指向具体子类的一个实例。抽象类的子类为父类中的所有抽象方法提供具体的实现，否则它们也是抽象类。

接口可以被看作抽象类的变体。接口中的所有方法都是抽象的，可以通过接口来间接地实现多重继承。接口中的成员变量都是static final类型。由于抽象类可以包含部分方法的实现，因此在一些场合下抽象类比接口存在更多的优势。

相同点：

  1. 都不能被实例化。
  2. 接口的实现类或抽象类的子类都只有实现了接口或抽象中的方法后才能被实例化。

不同点：

  1. 接口只有定义，其方法不能在接口中实现，只有实现接口的类才能实现接口中定义的方法，而抽象类可以有定义与实现，即其方法可以在抽象类中被实现。
  2. 接口需要实现，但抽象类只能被继承。一个类可以实现多个接口，但一个类只能继承一个抽象类，因此我们使用接口可以间接的实现多重继承的目的。
  3. 接口强调特定功能的实现，其设计理念是“has-a”关系；而抽象类强调所属类关系，其设计理念为“is-a”关系。
  4. 接口中定义的成员变量默认为public static final，只能够有静态的不能被修改的数据成员，而且必须给其赋初值，其所有成员方法都是public、abstract的，而且只能被这两个关键字修饰。而抽象类可以有自己的数据成员变量，也可以有非抽象的成员方法。
  5. 接口被用于实现比较常用的功能，便于日后维护或者添加删除方法；而抽象类更倾向于充当公共类的角色，不适用于日后重新对里面的代码进行修改。
  • 抽象类
  • 接口

内部类

在Java语言中，可以把一个类定义到另外一个类的内部，在类里面的这个类就叫做内部类，外面的类叫外部类。在种种情况下，这个内部类可以被看作外部类的一个成员（与类的属性和方法类似）。

内部类主要有四种：静态内部类、成员内部类、局部内部类和匿名内部类。

class outerClass{
    static class innerClass{}    //静态内部类
}
class outerClass{
    class innerClass{}        //成员内部类
}
class outerClass{
    public void memberFunction(){
        class innerClass{}        //局部内部类
    }
}
class outerClass{
    Handler handler = new Handler(){    //匿名内部类
        //.......
    };
}

静态内部类是指被声明为static的内部类，它可以不依赖于外部类实现而被实例化，通常的内部类需要在外部类实例化后才能实例化。静态内部类不能与外部类有相同的名字，不能访问外部类的普通成员变量，只能访问外部类中的静态变量和静态方法（包括私有类型）。

一个静态内部类如果去掉static关键字，就成为成员内部类，它可以自由地引用外部类的属性和方法，无论这些方法和属性是否是静态的。成员内部类中不可以定义静态的属性和方法。

局部内部类是指定义在一个代码块内的类，它的作用范围是其所在的代码块，是内部类中使用最少的一种类型。局部内部类像局部变量一样，不能被public、protected、private及static修饰，只能访问方法中定义为final类型的局部变量。

匿名内部类是一种没有类名的内部类，不使用关键字class、extends、implements，没有构造函数,它必须继承其他类或实现其他接口。

  1. 匿名内部类不能有构造函数。
  2. 匿名内部类不能定义静态成员、方法和类。
  3. 匿名内部类不能是public、protected、private、static。
  4. 只能创建匿名内部类的一个实例。
  5. 一个匿名内部类一定是在new的后面，这个匿名类必须继承一个父类或实现一个接口。
  6. 因为匿名内部类为局部内部类，所以局部内部类的所有限制都对其有效。

static关键字

static关键字主要有两种作用

  1. 为某特定数据类型或对象分配单一的存储空间，而与创建对象的个数无关。
  2. 实现某个方法或属性与类而不是对象关联在一起，也就是说在不创建对象的情况下就可以通过类来直接调用方法或使用类的属性。

static成员变量

Java语言没有全局的概念，但可以通过static关键字来达到全局的效果。用static关键字修饰的静态变量属于类，在内存中只有一个复制，只要静态变量所在的类被加载，这个静态变量就会被分配空间，对静态变量的两种引用方式分别是“类.静态变量”和“对象.静态变量”。

静态变量只有一个，被类拥有，所有对象都共享这个静态变量，需要注意的是，不能在方法体中定义static变量。

static成员方法

static方法是类的方法，不需要创建对象就可以被调用，而非static方法是对象的方法，只有对象被创建出来后才可以被使用。

static方法中不能使用this和super关键字，不能调用非static方法，只能访问所属类的静态成员变量和成员方法，因为当static方法被调用时，这个类的对象可能还没被创建。

static代码块

static代码块在类中是独立于成员变量和成员函数的代码块的，它不在任何一个方法体内，JVM在加载类时会执行static代码块，如果有多个static代码块，JVM将会按顺序来执行。static代码块通常被用来初始化静态变量。这些代码块只会被执行一次。

static内部类

static内部类是指被声明为static的内部类，它可以不依赖于外部类实例对象而被实例化，而通常的内部类需要在外部类实例化后才能实例化。静态内部类不能与外部类有相同的名字，不能访问外部类的普通成员变量，只能访问外部类中的静态成员和静态方法。只有内部类才能被定义static。

《增长黑客》1：什么样的人是增长黑客？

发布自简书-浅斟意

“增长黑客”定义：

是一群以数据驱动营销、以市场指导产品，通过技术化手段贯彻增长目标的人。他们是介于技术和市场之间的新型团队角色，通常既了解技术，又深谙用户心理，擅长发挥创意、绕过限制，通过低成本的手段解决初创公司产品早期的增长问题。在外行人眼里，他们就像是极客、发明家和广告狂人的混合体 。

增长黑客的职责：

  1. 数据分析——增长的工作建立在对大量数据的分析衡量基础之上。
  2. 用户获取，如SEO，PPC，EDM
  3. 指导产品研发
  4. 促进（数据）文化建设
  5. 增长快，对人才招聘有好处

增长黑客的一般工作流程：

发现问题-提出可能解决问题的假设-围绕假设设计试验方案并得出结论-将成功的试验结论固定化及产品化，形成长效增长机制。
