产品设计思考(1)：我为什么要引入拾光奖学金模块

在校园内“奖学金活动”沉淀用户成功，最终我们将此活动沉淀在产品模块中。
——这是运营推动产品的一个案例。

为什么设计拾光奖学金模块？
  1. 用原欲激励用户，提高用户自控力和毅力，提高产品留存率。
    产品设计：App里有各种奖学金挑战，完成挑战并且排在前几名的就有奖金。
    在我们对考研人群的访问中，我们发现，他们有很强烈的需求，但由于自控能力不强。用时间管理软件并不难很好地帮助他们放下手机。现在市面上很多时间管理软件，都是在“教导”用户：要理性地控制自己的欲望来提高效率。可是由于它们反人性，很少有人坚持使用，更不要说能够有引爆点。
    同样是一款时间管理软件，为什么我们很自信拾光能够引发爆炸性传播呢？
    因为我们顺应人性。我们的主题是：时间就是金钱——它意味着，你用了拾光，不仅能够屏蔽信息打扰、安心做事，你还很有可能获得金钱鼓励。有钱谁不要啊！而且我也在做一件正确理性的事情，挑战自己不玩手机啊！就这样，我们把时间管理变成一种挑战，而且是一种能够赚钱的挑战。不用理性教导，而是通过人的原始欲望（赢取奖金）打败原始欲望（玩手机）。通过奖励，提高用户留存率。
    这个模式已经问过很多研友，他们纷纷表示，愿意用！有钱又帮我，为啥不用？！
  2. 正向激励，引发传播，引爆App。
    产品设计：利用规则引发传播——越多人用，奖金额度越高。
    挑战金额根据参与人数而定，引发用户传播挑战；挑战存在正向激励，能够引发自豪感，完成任务后，引发用户炫耀结果；参与挑战的伙伴间相互督促，让这件事变得更有趣、有互动性，而不再是一个人的事。
如何设计规则，提高自传播？

那么，我们怎么设计奖金额度，使它可以达到爆炸性传播呢？

规则很重要。

首先，我们的奖金额度不是固定的，是根据参与的人数决定的。我设立一个系数，奖金总额=系数*人数。

那么作为参与者，我就希望越多人参与越好，我就可能得到越多钱啊！

这就促进了这个软件在用户间自发地传播，并且用户在给别人介绍的时候，也是一种对朋友的慷慨。双方还可设置监督人来帮助对方锁屏。

其次，这个参与挑战的申请是限时的，以此促进大家转发。

奖金从哪里来？

引入商家，合作共赢。

分析：现在很多的商家需要推广，他们把钱花在了赞助各种活动上，却得到很少的回报和曝光率。而我们的拾光奖学金，为他们提供了一个推广的渠道。

相比传统推广渠道，拾光奖学金能够给商家的品牌曝光带来的好处有：

  • 正向反馈：拾光用户越多，商家的曝光率越高，商家更乐意提供奖学金；而奖学金越多，拾光用户会越多。
  • 宣传时间长：奖学金有持续时间，商家可以持续曝光；
  • 品牌形象：冠名奖学金“逼格”很高，能够提高企业的品牌形象和知名度；
  • 宣传地域广：打破了区域的限制，能把品牌推广做向全国。

这满足了商家对于校园推广的极大需求。

后期我们甚至可以引入培训机构的现金券；甚至小企业的一顿饭等（针对2-4个人左右的挑战任务）。

这个模式很新，我们相信他能够引爆，使得三方多赢；而因为奖学金模块，我们也能够有现金流，实现快速盈利。

效果评估？

此模块确实提高了留存率，但没有引爆。

原因：产品没有提供支持，在第一次活动时，甚至没有转发按钮；

改进：后期产品进行了改进，引入分享按钮。同时，在期末复习期间，联合校园八大媒体开展“告别手机挑战倒计时”活动，曝光率高，新增用户量及用户活跃度有了明显的提升。

ASO之安卓和IOS优化方案

总结自公众号【姑婆那些事儿】ASO优化系列，并融入自己的思考——

安卓类商店的ASO优化

安卓的商店比较多，有的商店后台提供提交关键词的功能，有的则不能；有的在应用描述中的文字有权重，有的没有。

整体优化思路为：

  • 覆盖热词——通过百度指数/热度搜索等找流量高（流量大小参考搜索热词的热度appying、百度指数等）、竞争难度低的词，在后台提交关键词（这个权重大，但字数限制）/应用描述文字（安卓应用商店产品描述中有权重的商店有：豌豆荚、360、安卓市场、小米等；字数不限，发挥余地大）；
  • 优化下载量、评论（花钱，评论一般1元一条。下载量一般都是几十块钱一万下载量；如果要保证ASO效果的话，下载量建议持续做至于一次做多少量，能达到效果，还得自己摸索。）效果验证可以通过酷传（http://www.coolchuan.com）的应用监控功能查看。

小雨：拾光目前主流应用市场采用apk直接上传，其他市场采用酷传。

Object中公有方法

参考书籍：《Java核心技术》、《Java编程思想》

Obeject是所有类的超类，java是一门单根继承的程序语言。所有类都是由Object类扩展而来的。

equals方法

用于检测一个对象是否等于另一个对象。在Object类中，这个方法用于判断两个对象是否具有相同的引用。如果两个对象具有相同的引用，它们一定是相等的。

下面看一个例子：

class Test{
    int index;
}

public class TestEqual{
    public static void main(String[] args){
        Test test1 = new Test();
        Test test2 = new Test();
        System.out.print(test1.equals(test2));
    }
}

这个测试的结果是false。这是因为equals方法是比较引用的，所以除非在自己的新类中覆盖equals方法，否则不可能实现出相等的结果。

Java中要求equals方法具有下面的特性：

  1. 自反性：对于任何非空引用x，x.equals(x)应该返回true。
  2. 对称性：对于任何引用x和y，当且仅当y.equals(x)返回true，x.equals(y)也应该返回true。
  3. 传递性：对于任何引用x、y和z，如果x.equals(y)返回true，y.equals(z)返回true，x.equals(z)也应该返回true。
  4. 一致性：如果x和y引用的对象没有发生变化，反复调用x.equals(y)应该返回同样地结果。
  5. 对于任意非空引用x，x.equals(null)应该返回false。

我们以后再来看如何在自定义类中编写合适的equals方法。

hashCode方法

散列码（hash code）是由对象导出的一个整型值。散列码是没有规律的。如果x和y是两个不同的对象，x.hashCode()和y.hashCode()基本上不会相同。

Equals和hashCode的定义必须一致：如果x.equals(y)返回true，那么x.hashcode()就必须与y.hashcode()具有相同的值。

toString方法

它用于返回表示对象的字符串。只要对象与一个字符串通过操作符“+”连接起来，Java编译就会自动地调用toString方法，以便获得这个对象的字符串描述。每一个非基本类型的对象都有一个toString方法，而且当编译器需要一个String而只有一个对象时，该方法便会得到调用。

这里只是对Object中一些公用方法的简单介绍，更详细的介绍我们以后在讨论。

两个小问题

今天来看两个小问题。

switch能否用String作为参数

case标签必须是整数或枚举常量，不能测试字符串。例如，下面这段代码就存在错误。

String input = ...;
switch(input){ //ERROR
    case "A": //ERROR
        ...
        break;
    ...
}

但是从Java SE 7开始，case标签还可以是字符串字面量。也就是像上面的写法在Java SE 7中是没有问题的。

equals与==的区别

检测字符串是否相等可以使equals方法检测两个字符串是否相等。对于表达式：

s.equals(t)

如果字符串s与字符串t相等，则返回true；否则，返回false。

要想检测两个字符串是否相等，而不区分大小写，可以使用equalsIgnoreCase方法。

一定不能使用==运算符检测两个字符串是否相等！这个运算符只能够确定两个字符串是否放在同一个位置，它们必然相等。但是，完全有可能将内容相同的多个字符串的拷贝放置在不同的位置上。（引用）

String greeting = "Hello";
if(greeting == "Hello") ... //probably true
if(greeting.substring(0,3) == "Hel") ... probably false

我们再看一个例子

public class Test{
    public static void main(String[] args){
        Integer n1 = new Integer(47);
        Integer n2 = new Integer(47);
        System.out.println(n1 == n2);
    }
}

这段代码的输出是false。这是因为==比较的是对象的引用。那么如果我们想比较两个对象的实际内容是否相同，就必须使用特殊方法equals()。但是该方法如果直接在自定义类中调用那就又有很多问题。以后我们会讨论。
