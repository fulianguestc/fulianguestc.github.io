Java中this的用法

最初接触Java时，就一直无法理解this的用法，觉得很多地方都有它的身影。下面就简单介绍一下this的集中基本用法。

主要参考书籍《Java核心技术.卷1》。

1. 隐式参数与显示参数

方法用于操作对象以及存取它们的实力域。例如，方法：

public void raiseSalary(double byPercent)
{
    double raise = salary * byPercent / 100;
    salary += raise;
}

raiseSalary方法有两个参数。第一个参数被称为隐式参数，是出现在方法名前的Employee类(本方法所在类)对象。第二个参数位于方法名后面括号中的数值，这是一个显式参数。

显示参数是明显地列在方法声明中的的显示参数，例如double byparent。隐式参数没有出现在方法声明中。

关于adb-shell的若干问题

最近一直在进行安卓开发，在windows下启动adb shell只要配置环境变量就好了，并且网上教程很多，那么在MAC下如何配置呢？

大体思路是一致的，只是要用命令行来操。

首先启动终端，然后输入：

$ touch .bash_profile
$ open -e .bash_profile

这时会打开.bash_profile这个文档,

export PATH=${PATH}:/Users/***/sdk/platform-tools

在后面输入platform-tools所在的目录路径，比如笔者的路径就是/Users/***/sdk/platform-tools，所以写成上面的形式。然后直接关闭文档自动保存。

软件开发之后……

软件开发后的常规流程：

  1. 内测。
    软件测试完成，进行内测（周边朋友内测，固定合作组织机构内测，部分小市场投放内测）
  2. 产品优化。
    内测完成，将用户反馈收集起来进行修改调整，同步进行宣传资料的准备，联系广告推广上，合作完成一部分软件投放设计。
  3. 市场运营推广。
    进入市场投放（校园实地宣传、网络软文宣传、社交平台宣传、合作组织推广宣传）
  4. 产品优化。
    投放市场后跟踪市场状况，实时推出新的信息，跟进宣传。同时收集用户反馈，进行产品优化，了解用户新需求，寻求盈利点。
  5. 盈利探索。
    完善软件，增加增值服务点，寻求盈利。

产品设计思考(2)：拾光挑战是否可以同时进行

这篇是我关于拾光产品挑战模块的思考。

【挑战模块设计】为什么要设计挑战模块？

用户对挑战项目的参与度很高（如同样反人性的戒客），可以考虑设计长期挑战/关卡。不同等级可以参与的挑战不同，引导用户积极提升等级，提高活跃度。

产品设计思考(1)：我为什么要引入拾光奖学金模块

在校园内“奖学金活动”沉淀用户成功，最终我们将此活动沉淀在产品模块中。
——这是运营推动产品的一个案例。

为什么设计拾光奖学金模块？
  1. 用原欲激励用户，提高用户自控力和毅力，提高产品留存率。
    产品设计：App里有各种奖学金挑战，完成挑战并且排在前几名的就有奖金。
    在我们对考研人群的访问中，我们发现，他们有很强烈的需求，但由于自控能力不强。用时间管理软件并不难很好地帮助他们放下手机。现在市面上很多时间管理软件，都是在“教导”用户：要理性地控制自己的欲望来提高效率。可是由于它们反人性，很少有人坚持使用，更不要说能够有引爆点。
    同样是一款时间管理软件，为什么我们很自信拾光能够引发爆炸性传播呢？

ASO之玩转积分墙

总结自公众号【姑婆那些事儿】ASO优化系列，并融入自己的思考——

之前讨论过热词覆盖：

能带来有效流量的词是要保留的，有效的关键词的热度不一定高，热度高的关键词不一定流量最大，这里需要做一个取舍，主要看该词是否和你的App适配。（热度高，又与你的app相关，则流量大）。

如何检验一个词带来多少量呢，比如，一个“约炮app”，做“交友”这个关键词，如果你做到第一了，维持7天左右时间，算出平均新增，减去之前未做ASO时候的新增，方法笨，但实用，准确。前提是渠道比较稳定——有些产品渠道比较多，新增也不稳定，可能这个方法不一定适用。
